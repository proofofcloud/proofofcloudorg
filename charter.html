Draft Charter v0.1 for the Proof of Cloud Alliance

Draft Charter v0.1

Purpose

Proof of Cloud (“PoC”) is a vendor-neutral alliance that maintains a signed, append-only registry of cloud-hosted server hardware identities (e.g., Intel SGX/TDX PPIDs, AMD SEV-SNP chip IDs), along with the evidence and endorsements establishing each entry’s verification level. The registry’s goal is to make it verifiable—not merely asserted—that a workload runs on genuine cloud hardware at a legitimate facility, while being explicit about residual risks and non-goals.

1. Scope & Non-Goals

  • Scope: Define verification levels and procedures for adding server identities to a public, signed registry; publish periodic, co-signed snapshots and a revocation list.
  • Non-Goals: The alliance does not claim protection against all physical attacks or certify the full operational security of any provider. It does not coordinate pricing, capacity, or other market behaviors (antitrust safe harbor).

2. Governance

  • Members: General members only. One company = one member = one vote.
  • Chair: Elected for 6-month terms; sets agendas, moderates votes, and can act as tie-breaker. (Optional vice-chair.)
  • Quorum & Voting: Simple majority quorum (with 8 members, 5 present). Decisions pass by simple majority unless security-critical.
  • Security-critical changes: Schema/protocol changes and emergency revocations require ≥ 2/3 of all members.

3. Working Groups

  1. Verification WG: Daily onboarding, evidence review, revocation proposals.
  2. Methodologies WG: Maintains verification methods, proof templates (zkTLS/TLSNotary), threat models, schemas.
  3. Ops & Outreach WG: Website, registry infra, keys & signatures, snapshots, DNS anchoring, docs, community.

4. Verification Levels

L1 — Member mutual verification (manual + Remote Attestation)

Two independent members perform a live verification: run hardware RA (Intel DCAP SGX/TDX or AMD SEV-SNP), extract the identity (PPID/Chip ID), and capture a cloud instance identity token where available (AWS IID / GCP OIDC / Azure IMDS). Bind them with a nonce. In case the hardware is in Colocation, members perform a verification of signed Colocation Provider Pledge. Produce a signed evidence pack, with a 30–60 day validity window.

L2 — API-bound + zkTLS, with continuous monitoring

Automate retrieval of cloud control-plane identity and bind it to RA (nonces → RA user data). Capture verifiable TLS transcripts (TLSNotary/zkTLS-style) proving what the CSP API stated at time t. Add weekly (or better) re-attestation, TCB-change alerts, and drift detection.

L3 — Cloud-issued bound certificates (future)

The cloud vendor issues a signed certificate that directly binds instance/host identity to PPID/Chip ID. Registry can then verify and include entries automatically.

5. Registry & Privacy

  • Entry fields:
    • platform_uid (e.g., intel:tdx:ppid:..., amd:snp:chipid:...)
    • level (1–3)
    • evidence[] (typed, hashed)
    • monitoring (cadence/last/next)
    • endorsements[]
    • status
    • timestamps
  • Public views: Default to salted hashes of PPIDs/Chip IDs; full values visible to members for legitimate security purposes.
  • Snapshots: Append-only Merkle tree; periodic co-signed snapshots; anchor in DNS (optional on-chain).

6. Incidents & Amendments

  • Flagging SLA
    • Verification WG triages within 72h.
    • Emergency quorum may freeze an entry.
  • Amendments
    • Editorial: simple majority.
    • Security-relevant: 2/3.
